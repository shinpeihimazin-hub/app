# price-hunt: 商品最安値リサーチスキル

ユーザーが欲しい商品の「正解（＝これを買うべき、という結論）」に至るまで、曖昧な要望から型番特定・複数サイト比較・検証・提示までを一気通貫でやり切るためのスキル。手を抜いて1〜2サイトだけ見て終わらせない。

このスキルの核心は「表示価格の単純比較」ではなく、**楽天SPU・5と0のつく日・お買い物マラソン、Yahoo!ショッピングの5のつく日・PayPay還元、Amazonのポイントアップキャンペーンなど、日本のECポイント制度をユーザー個別の条件まで踏み込んで計算し、真の実質価格で結論を出すこと**にある。ポイント制度の詳細・重なり方は [`points-knowledge.md`](points-knowledge.md) を必ず参照する。ここを省略した比較は価格.comの単純な価格順ソートと同じで、このスキルを使う意味がない。

## 進め方（6ステップ）

### 0. ポイント環境ヒアリング（最重要・省略しない）
`user-profile.md` が存在する場合はまずそれを読み込み、記載済みの項目は再度聞かない。存在しない、または一部項目が未記入の場合のみ、`points-knowledge.md` の「ヒアリング項目」に沿ってユーザーのポイント環境を確認する。これを省略すると、後段の実質価格計算がユーザー個別の正解にならず、一般論の概算に落ちる。

- 主に使うクレジットカードと還元率
- 楽天SPU対象サービスの加入状況（楽天モバイル・楽天カード・楽天銀行など）
- Yahoo!ショッピング/PayPay利用状況、LYPプレミアム加入有無
- 各モールの今月のポイント獲得上限消化状況（わかる範囲で）
- 購入予定日（5と0のつく日、お買い物マラソン開催中かなど）
- 獲得ポイントを実際に使い切れる見込み（期間限定ポイントの実質価値判定用）
- アカウントを持っているポイントサイト（モッピー、ハピタス、ECナビ、ポイントインカム、楽天リーベイツなど）

新しく聞き出した内容、または既存の`user-profile.md`から変更があった内容は、比較が終わったあとに`user-profile.md`へ書き戻し、次回以降は再ヒアリングを省略できるようにする。ユーザーが即答できない項目は「一般的な還元率で概算」であることを明示した上で先に進めてよいが、最終提示時に「この項目が未確認のため概算です」と必ず注記する。

### 1. 商品特定フェーズ
ユーザーの要望が曖昧な場合（例:「軽くて防水のワイヤレスイヤホン」）は、購入の決め手になる条件を質問して型番・型式まで絞り込む。すでに型番が分かっている場合はこのステップを飛ばす。

確認すべき軸の例:
- 用途・優先条件（音質、バッテリー、価格帯など）
- 色・容量・世代など選択肢が分かれる仕様
- 新品/中古/整備品の許容範囲
- 国内正規品が必須か、並行輸入可か

型番・正式名称が確定するまで検索フェーズに進まない。ユーザーが「なんでもいいから探して」と言った場合のみ、最も無難な標準構成で先に進めてよい。

### 2. 価格収集フェーズ
型番が確定したら、WebSearch / WebFetch を使って `target-sites.md` に登録されたサイトを横断的に調べる。登録数に技術的な上限はない（API連携ではなくWebSearch/WebFetchで都度調べる方式のため）。デフォルトでは最低でも以下のカテゴリを調べ、1サイトだけの結果で終わらせない。ユーザーが個別サイトを追加したい場合は `target-sites.md` に追記して恒久的に対象へ加える。

- 大手ECモール: Amazon、楽天市場、Yahoo!ショッピング
- カード連携モール: `user-profile.md`の保有カード一覧と`target-sites.md`の「カード連携モール」を必ず突き合わせ、該当するモール（au PAY マーケット、JR Eモール、イオンスタイルオンラインなど）を漏らさず調査対象に加える。保有カードに対応するモールを見落とすのはこのスキルの信頼性に直結するミスなので、価格収集を始める前に必ず突き合わせを行う
- 比較サイト: 価格.com、コジマ×ビックカメラ等の価格比較エンジン
- メーカー直販・公式オンラインストア
- 家電量販店のオンラインストア（ヨドバシ.com など、該当カテゴリなら）
- 中古/整備品市場（ユーザーが許容していれば：メルカリ、イオシス等）

各サイトについて記録する情報:
- 商品名・型番が完全一致しているか（型落ち品やOEM品を混同しない）
- 本体価格
- 送料
- ポイント還元率・還元額（実質価格の計算に使う）
- クーポン・セール情報の有無と適用条件
- 在庫状況
- 販売元（公式 or 第三者マーケットプレイス出品者か）

### 3. 実質価格の正規化（ポイント制度を織り込む本丸）
サイトごとにバラバラな条件を、ステップ0で聞いたユーザーのポイント環境に基づいて揃える。`points-knowledge.md` の各モールの仕組みと照らし合わせて、以下を必ず個別に計算する。

実質価格 = 本体価格 + 送料 − 通常ポイント還元 − 上乗せキャンペーン還元（SPU/買い回り/5と0の日/5のつく日など、ユーザーが対象条件を満たす分のみ）− 即時適用クーポン − クレジットカード還元

計算時の注意:
- ユーザーが対象条件（SPU加入サービス、決済日、エントリー状況）を満たしていない上乗せ分は加算しない
- キャンペーンに月間獲得上限がある場合、その商品購入でユーザーの上限に到達するかを確認し、到達するなら超過分は還元額から除く
- 期間限定ポイントで「使い切れる見込みが低い」とユーザーが答えた場合は、実質価値を割り引く（半額査定または加算しないなど、判断根拠を明示した上で控えめに見積もる）
- ポイント還元率や各キャンペーンの現在の条件はWebSearchで当日時点の公式情報を確認する。古い記憶の数値を断定的に使わない

ポイントの実質価値が不明瞭な場合（期間限定ポイント、特定店舗限定など）は、実質価格に加算せず別枠で注記する。

さらに、ユーザーが保有しているポイントサイト（モッピー、ハピタス、楽天リーベイツなど）経由でそのストアにアクセスできる場合、`points-knowledge.md`の「ポイントサイト経由」に従って上乗せ還元をWebSearchで確認し、最終的な実質価格から差し引く。未保有のポイントサイトなら「新規登録すればさらに◯円下がる」という参考情報に留め、保有分だけを結論の実質価格に含める。

### 4. 検証フェーズ（誤情報・詐欺サイトの排除）
以下に該当するものは候補から除外するか、明確に警告した上で提示する。

- 型番・型式が微妙に異なる別モデル（型落ち、地域限定モデル、簡易版など）
- 相場から極端に乖離した安値（詐欺サイトの可能性）
- 販売実績・レビューが極端に少ない/不自然な第三者出品者
- 在庫切れ・予約商品なのに通常価格で表示されているもの
- 公式サイト以外で「正規品」を明言していないのに正規品として扱われているもの

### 5. 結果提示フェーズ
以下の形式で、根拠を示しながら結論を出す。

1. 一番安い実質価格の選択肢と、その根拠（サイト名・本体価格・送料・通常ポイント・上乗せキャンペーン・クーポン・クレカ還元・ポイントサイト経由分の内訳・URL、経由が必要なら「まず◯◯（ポイントサイト）のリンクを踏んでから購入」という手順も明記）
2. 次点2〜3件との比較表（本体価格・送料・還元内訳・実質価格・販売元）
3. 除外した怪しい候補があれば、除外理由を簡潔に記載
4. ステップ0で前提とした購入日・ポイント環境（未確認だった項目があればそれも）を明記し、条件が変わると結論が変わりうることを伝える
5. 型番特定時に前提とした条件（ステップ1の確認内容）を最後に再掲し、認識齟齬がないか確認する

## 注意事項

- 実在しないURLやサイトをでっち上げない。WebSearch/WebFetchで実際に確認できた情報のみを使う。
- 価格情報は変動するため、いつ時点の情報かを明記する。
- 個人情報やアカウント情報の入力が必要な操作（購入確定など）はユーザー自身に行ってもらい、代行しない。
