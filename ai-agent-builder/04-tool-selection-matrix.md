# フェーズ3: ツール選定マトリクス

各タスクについて「Claudeで全部やる」を初期解にせず、最適な言語・フレームワーク・モデル・ツールを毎回比較して選ぶ。

このファイルの原則: **個別ツール名は変わる。選定の「軸」を先に固定する。** ツール名は例示であり、案件時点での最新版・代替を必ず確認する（このキットは特定バージョンに依存しない）。

## 先に読む: プリミティブ選定は別ファイル

「そもそもこの処理を、スキル／サブエージェント／スラッシュコマンド／MCP／プラグイン／Agent SDK のどれで作るか」という**エージェント・プリミティブの選定**は、このマトリクスの前段にある最重要判断で、[`08-agent-primitives-and-composition.md`](./08-agent-primitives-and-composition.md) に集約している。フェーズ3.5として必ず先に通す。このファイル（カテゴリ1〜9）は、プリミティブを決めた後の「中身をどのモデル・FW・DB・基盤で実装するか」の選定に使う。

**サードパーティの具体候補カタログは [`18-third-party-ecosystem.md`](./18-third-party-ecosystem.md)**: エージェントのモダリティ別ツール（ブラウザ操作/音声/computer use）、既製MCPサーバー群、追加フレームワーク（Pydantic AI/Google ADK/Microsoft Agent Framework/Mastra等）、専門ツール（検索API/コード実行サンドボックス/メモリレイヤー/ドキュメント解析）、A2Aプロトコル。このファイルの軸で、18の候補を評価する関係。

---

## 大原則: 選定は「軸」で決める

どのカテゴリでも、以下の共通軸でスコアリングして選ぶ。フェーズ0で確定した優先順位（速度/精度/コスト/保守性/拡張性）に応じて軸の重みを変える。

- 実装コスト（初期工数）
- 運用コスト（課金・保守・人件費）
- 柔軟性・拡張性
- 学習コスト・チームの既存スキル
- エコシステムの成熟度・実績・ドキュメント
- 機密情報・コンプライアンス制約との整合
- ロックインの度合い（乗り換えやすさ）

## カテゴリ1: LLMモデル選定

**判断の起点は2つ: ①機密データを外部APIに出してよいか ②精度とコストのどちらを優先するか。**

| 選択肢 | 向いているケース | 注意点 |
|---|---|---|
| 高性能クローズドモデル（Claude / GPT / Gemini の上位モデル） | 精度優先、複雑な推論・長文脈・エージェント制御が要る | トークン課金。機密データの外部送信可否を要確認 |
| 軽量クローズドモデル（各社の小型・高速モデル） | 速度・コスト優先、分類・抽出・単純タスク | 複雑な推論は苦手。タスクを絞る |
| オープンウェイトモデル（Llama / Mistral / Qwen 等をセルフホスト or 推論API経由） | 機密データを自社内に留めたい、大量処理でコストを抑えたい、カスタマイズ・ファインチューニングしたい | セルフホストは運用工数大。GPU確保・推論最適化が要る |
| タスク別使い分け（ルーティング） | コストと精度を両立したい | 「簡単なタスクは軽量モデル、難しいものだけ高性能モデル」に振り分ける実装が要る |

**選定のコツ**: まず一番高性能なモデルで動くものを作り、精度を確認してから、コストのためにより軽いモデルへ落とせるか検証する（逆順は精度不足の原因切り分けが難しい）。Claudeで作る場合のモデル選定・最新IDは、Claude Code環境なら `claude-api` スキルを参照して最新情報で確認する。

## カテゴリ2: オーケストレーションフレームワーク

**「そもそもフレームワークが要るか」から問う。** 単純な線形パイプラインなら、フレームワークなしで各ベンダーの公式SDKを直接叩くのが最も保守しやすいことが多い。

| 選択肢 | 向いているケース | 注意点 |
|---|---|---|
| 素のSDK直叩き（各ベンダーのSDK + 自前の制御コード） | 線形/単純ReAct、依存を増やしたくない、挙動を完全に把握したい | 複雑な状態管理・分岐は自分で書く |
| Claude Agent SDK（Claude Codeの基盤。ツール実行・権限・サブエージェント込み） | Claudeでコーディング・ファイル操作・ツール実行を伴う自律エージェントを作る | Claude前提 |
| グラフ型オーケストレーション（LangGraph 等） | 複雑な分岐・ループ・状態遷移・Human-in-the-Loopを明示的に制御したい | 学習コストあり。設計を要する |
| ロール型マルチエージェント（CrewAI / AutoGen / OpenAI Agents SDK 等） | 役割分担した複数エージェントの協調を素早く組みたい | 抽象度が高く、細かい制御やデバッグがしづらい場合がある |
| 統合フレームワーク（LlamaIndex / Haystack 等、RAG強め） | RAG中心の構成を手早く作りたい | エージェント制御は他FWの方が柔軟なことも |

**選定のコツ**: フレームワークはロックインと学習コストを生む。フェーズ1で判定したパターンが「線形」なら、まずフレームワークなしを検討。複雑な状態遷移やマルチエージェントが本当に必要になってから重いフレームワークを入れる。

## カテゴリ3: ノーコード / ローコード

**エンジニアの工数を使わずに済むか、非エンジニアが保守できるかが判断軸。**

| 選択肢 | 向いているケース | 注意点 |
|---|---|---|
| ワークフロー自動化系（n8n / Make / Zapier 等） | SaaS間連携・トリガー駆動の自動化、非エンジニア保守 | 複雑なエージェント制御は不得手。ステップ増で可読性低下 |
| LLMアプリ構築系（Dify / Flowise / LangFlow 等） | RAGチャットボット・簡易エージェントをGUIで素早く | 細かいカスタマイズや大規模化で限界。ホスティング形態を確認 |
| クラウドベンダーのマネージド（各クラウドのエージェント/ボット基盤） | 既にそのクラウドを使っている、認証・スケールを任せたい | ベンダーロックイン |

**選定のコツ**: PoC・社内ツール・低頻度の自動化はノーコードが圧倒的に速い。本番の高負荷・複雑ロジック・独自要件が強いものはコード実装へ。「ノーコードで作って限界が来たらコードへ移す」前提の切り分けを最初に決めておく。

## カテゴリ4: RAG・ベクトルDB・検索

**まず「RAGが本当に要るか」を問う。** LLMの一般知識で足りるならRAGは作らない。長文コンテキストに全部入れて済むならそれでよい場合もある。

| 選択肢 | 向いているケース | 注意点 |
|---|---|---|
| コンテキスト直入れ（検索なし） | 参照文書が少量で毎回全部入る | 大量文書では不可・高コスト |
| キーワード/全文検索（既存の検索エンジン流用） | 完全一致・専門用語が重要 | 意味的な曖昧検索は弱い |
| ベクトル検索（Pinecone / Weaviate / Qdrant / Milvus / pgvector 等） | 意味的な類似検索、大量文書 | Embeddingモデル選定・チャンク設計・更新運用が要る |
| ハイブリッド検索（ベクトル+キーワード+リランク） | 精度を最大化したい | 構築・チューニング工数が増える |
| マネージドRAG（各ベンダーのマネージド検索/ファイル検索機能） | 運用を任せたい、素早く立ち上げたい | カスタマイズ性・コストを確認 |

**選定のコツ**: RDBを既に使っているなら、まず pgvector 等で既存DBに載せられないか検討する（インフラを増やさない）。規模・レイテンシ要件が専用ベクトルDBを要求する段階で移行する。

## カテゴリ5: ツール実行・外部連携（MCP含む）

| 選択肢 | 向いているケース | 注意点 |
|---|---|---|
| ベンダー固有のFunction/Tool calling | 単一モデル・少数ツール | ベンダーごとに定義形式が異なる |
| MCP（Model Context Protocol）でツールをサーバー化 | ツールを複数エージェント/複数クライアントで再利用したい、標準化したい | サーバー実装・運用が増える。既存MCPサーバーの流用も検討 |
| 自前のツールラッパー（社内APIを関数化） | 社内固有システムと繋ぐ | 認証・エラー処理・レート制限対応を自作 |

**選定のコツ**: ツールを1つのエージェントでしか使わないならベンダー固有のTool callingで十分。複数エージェント・複数プロダクトで同じツールを再利用する見込みがあるならMCP化を検討。

---

## カテゴリ6: メモリ・状態管理

| 選択肢 | 向いているケース | 注意点 |
|---|---|---|
| ステートレス（毎回入力だけで完結） | 単発タスク | 会話の継続・学習は不可 |
| 会話履歴の保持（セッション単位） | チャット的な連続対話 | 長くなるとトークン肥大。要約・トリミング戦略が要る |
| 永続メモリ（DB/KVSに保存、ユーザー横断） | ユーザーの好み・過去のやり取りを跨いで参照 | プライバシー・保持期間・削除要求への対応が要る |
| 専用メモリレイヤー（Mem0 等のメモリ管理ツール） | メモリ管理を作り込みたくない | 依存とコストが増える |

---

## カテゴリ7: 評価・可観測性

**「評価なしのエージェントは本番に出さない」。** ここを削るとフェーズ5が回らない。

| 選択肢 | 向いているケース | 注意点 |
|---|---|---|
| 自前の簡易評価スクリプト | 小規模、指標が単純 | 大規模化で管理が破綻 |
| 評価・トレーシングプラットフォーム（LangSmith / Langfuse / Braintrust / Arize Phoenix 等） | 実行トレース・評価・回帰確認を継続的に回す | 導入・学習コスト。データ送信先の機密性を確認 |
| LLM-as-a-judge（LLMに出力の良し悪しを採点させる） | 正解が一意でない生成タスクの自動評価 | 採点者LLMのバイアス・コスト。人間評価との相関確認が要る |

詳細は [`06-evaluation-and-iteration.md`](./06-evaluation-and-iteration.md)。

## カテゴリ8: ガードレール・安全性

| 選択肢 | 向いているケース | 注意点 |
|---|---|---|
| プロンプト内制約のみ | 低リスク・社内利用 | 突破されうる。単独では不十分 |
| 入出力フィルタ（PII検出・禁止語・トピック制限。専用ライブラリ or 自作） | 個人情報・不適切出力を防ぐ | 過検出/検出漏れのチューニングが要る |
| Human-in-the-Loop承認 | 取り返しのつかない操作 | レイテンシとのトレードオフ |
| 権限・実行サンドボックス（ツール実行範囲の制限） | コード実行・ファイル操作・外部送信を伴う | 権限設計が要る |

---

## カテゴリ9: デプロイ・実行基盤

| 選択肢 | 向いているケース | 注意点 |
|---|---|---|
| ローカルCLI / スクリプト | 本人利用、開発中 | 共有・常時稼働は不可 |
| サーバーレス関数 | イベント駆動、低〜中頻度、スパイクあり | 長時間実行・状態保持に制約。コールドスタート |
| コンテナ常駐（自前サーバー/マネージドコンテナ） | 常時稼働、状態保持、高頻度 | 運用・スケール設計が要る |
| チャットプラットフォーム統合（Slack/Discord/Teamsボット） | チームで対話的に使う | プラットフォームのAPI制約・認証 |
| Claude Codeのスキル/サブエージェント | 開発ワークフロー内で使う、リポジトリ操作を伴う | Claude Code環境前提 |

## 選定アウトプットのフォーマット

各タスクごとに、最低2案を比較してこの形で出す。

```
### タスク: 社内文書からの根拠付き回答生成

| 案 | 実装コスト | 運用コスト | 拡張性 | 学習コスト | 実績 | 優先順位との整合 |
|---|---|---|---|---|---|---|
| A: pgvector + 素のSDK | 中 | 低 | 中 | 低 | 高 | コスト優先に合致 ◎ |
| B: 専用ベクトルDB + RAGフレームワーク | 高 | 中 | 高 | 中 | 高 | 拡張性は上だが今回は過剰 △ |

推奨: A
理由: フェーズ0で「コスト優先・既存PostgreSQLあり」と確定。文書数も数千件規模で専用ベクトルDBの性能は不要。インフラを増やさずに既存DBへ pgvector を載せる案Aが、運用コストと保守性で優る。将来 文書が数十万件規模になり検索レイテンシが要件を割った時点で案Bへ移行する。
```
