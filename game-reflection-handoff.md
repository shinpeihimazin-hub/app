> **重要な前提：戦略・行動設計・「ふり返りの本質」を詰めるのは “コア対話セッション”（呼び出し元）の役割。**
> このセッションは実装（プロト構築）担当。**設計の根幹に関わる判断は、勝手に決めずコア対話に戻すこと。**

## 0. このセッションのゴール

YouTube / Instagram を開いてしまう「空白の時間」に、**めちゃくちゃ気楽に遊べるカジュアルゲーム**を遊ぶと、
結果として **その日の振り返りが済んでいる** ―― そういうアプリの **最小プロト** を作る。

「完成品」ではなく「**箱庭プロトを小さく作って回す**」のが今回の到達目標。

---

## 1. 背景にある“本当のニーズ”（コア対話で確定済みの事実）

ユーザーの本当のニーズは、当初話に出ていた「瞑想／入眠」**ではない**（入眠は服薬タイミング変更で解決済み）。
確定している現在のニーズは:

1. **「積み上がっている」という実感が欲しい** ―― 日々やっているのに何も積み上がらない感覚を消したい。
2. **空白を埋めるだけの YouTube / Instagram をやめたい** ―― 「見たい物がある」訳ではなく、ただ時間を埋めている。もったいない。
3. **振り返って反省を次に活かすサイクル（C→A）** を回したい。それが回れば「もっと大きく動ける」はず。

### コア対話で判明した重要インサイト（=設計の土台。事実）

- **意思で勝つ／制限ツールで縛る は、両方このユーザーには効かない**（本人実績で証明済み。制限は都度解除してきた）。→ これらに依存する設計は禁止。
- YouTube/IG に逃げる瞬間の本人の状態 ＝ **「疲れている → でも何もしないのは耐えられない → 楽に刺激を入れたい」**。求めているのは“努力”ではなく**楽な刺激**。
- だから **燃料ゼロの時間に“努力”を要求する設計は必ず崩れる**（過去の崩壊の根本原因）。
- YouTube/IG が「もったいない」のは **痕跡がゼロ** だから。→ 戦略は「楽・刺激はそのまま、痕跡ゼロだけ差し替える」。
- **このユーザーの YT/IG には“中身の引力”がほぼ無い（空白埋め）**。→ 置き換えのハードルは見た目より低い＝勝ち目はある。

## 2. コンセプト（このアプリが狙うもの）

> **「逃げ込み先」を、我慢でなく“もっと気持ちいいもの”にして、そこにふり返りを忍ばせる。**

- 開くコストは IG 並み（ホーム画面から1タップ、起動即遊べる）。
- 1回の関与は **30秒〜数分**、気楽。考えさせ過ぎない。
- 遊ぶ＝**箱庭/ペット/世界が育つ**。この“育ち”が、コア対話で作った **「積み重ねの可視化（②痕跡）」をゲーム化したもの**。
- ふり返り（できた？／できなかった理由／今日の気づき）は、**ゲームの“行動”や“資源”として自然に発生**させる。「振り返り画面」を作業として出さない。

---

## 3. 設計原則・制約（=守らないと企画が死ぬ。最重要）

1. **気楽さ最優先。** 起動が重い・ロードが長い・チュートリアルが長い＝即死。IG に負ける。
2. **ふり返りを空洞化させない最低ライン**を必ず残す：
   - 「今日の一歩できた？」（done/not）
   - できなかったら **理由を“タップ”**（疲れた/重すぎた/忘れた/時間ない/気乗りせず）
   - 「今日の気づき」一行（任意・スキップ可）
   - ↑これらが、ゲームの1手（餌やり/水やり/1ガチャ 等）に化けている状態が理想。
3. **恥・罰でカスケードさせない。** 休んだ日に世界が**枯れすぎない／責めない**。一度サボると全部やめる人なので、復帰コストを限りなくゼロに。「おかえり」で戻れる設計。
4. **痕跡が残り、育つ。** 続けるほど世界が豊かになる（積み重ねが“見える”）。これが「積み上がってない実感」への直接の回答。
5. **意思力・制限に依存しない。** 「我慢して開く」ではなく「開きたいから開く」を狙う。通知スパム・強制ノルマで縛らない。
6. **“気楽 vs ふり返り”の両立が成否の核。** ここは**実装より先にプロトで検証**する論点。迷ったらコア対話に戻す。

## 4. 既存資産（再利用 or 連携できる）

現行アプリ `goal-manager.html`（GitHub Pages 配信の個人用PWA、`shinpeihimazin-hub/app`）に、コア対話で作った「最初の一歩」ループが既にある。**データモデルを流用 or 連携できる。**

localStorage キー（現行・事実）:
- `first-steps` … `{ "YYYY-MM-DD": { text, goalNodeId, done, doneTs } }` … その日の「最初の一歩」
- `reflections` … `{ "YYYY-MM-DD": { reason, note } }` … タップ振り返り（reason は固定5択 `["疲れてた","重すぎた","忘れてた","時間がなかった","気が乗らなかった"]`、note は気づき一行）
- `focus` … 文字列。「今の重点（積み上げたい一つ）」
- （旧データ：`goals` 目標ツリー、`task-list`、`routine-entries` 等も存在。今回は無視可）

→ **新ゲームは、この `first-steps` / `reflections` を読み書きする形にすれば、現行アプリと“同じ積み重ね”を共有できる**（同一オリジンなら localStorage 共有可能）。別アプリにするか、`goal-manager` 内の新タブにするかは未確定（§7）。

## 5. 推奨する「最初のプロト」（最小・叩き台）

> 目的は“面白い完成品”ではなく「**気楽さ × ふり返り** が両立するかの検証」。

例（あくまで一案・本人の題材選好で変わる §7）:

- **箱庭/育成系**（idle・低操作）。例：小さな庭／植木／ペット／島。
- 開くと、まず**今日の1手**：
  - 「今日の一歩、できた？」→ できた → 世界に“1つ咲く/育つ”＋小さな完了演出。
  - できなかった → 理由をタップ → 責めずに「そういう日もある」、世界は軽く待機（枯れない）。
  - 任意で「気づき」一行 → 入れると“ボーナス”的に何か育つ。
- **育ちの履歴＝積み重ねの可視化**（連続日数・累計・庭の成長）。
- 操作は最小（タップ中心）、1回30秒で閉じてOK。何度開いても気楽。
- 技術：**HTML5/JS（現行PWAの延長）でOK。** まずは1ファイルのプロトで可。重い3D・大規模アセットは不要（むしろ気楽さを殺す）。

**やってはいけない**：複雑なゲーム経済、長いチュートリアル、ふり返りのノルマ化、罰ゲーム化、通知連打、ログイン必須化。

## 6. 中核の未解決課題（正直に）

- **「気楽さ」と「ふり返りの中身」は本質的にトレードオフ**。タップだけで拾える振り返りは浅い。ここをゲームの“育ち”の意味付けでどこまで埋められるかは、**作って試さないと分からない**。最大の不確実性。
- カジュアルゲームは**飽きる**。育ちの content/進行をどこまで用意するかは継続課題。
- IG/YouTube の即時刺激に**生の強度では勝てない**。勝ち筋は「強度」ではなく「痕跡が残る満足＋空白埋め相手の引力ゼロ」。

1. **題材**：庭／植物／ペット／島／街／コレクション／キャラ育成 … **どれが本人に刺さって“開きたくなる”か**。← これは行動設計の核なので **コア対話で詰める**。

3. **プラットフォーム**：当面 PWA（GitHub Pages）継続でよいか。
4. **“開きたくなる”フック**：IG の代わりに指が伸びる理由を何にするか（育ちが見たい／続きが気になる／可愛い 等）。
5. **どこまでを「ふり返り」と呼ぶか**：done/理由/気づき の3点で十分か、もう少し欲しいか。

## 8. 実装セッションへの指示（最初の一手）

1. この資料を読み、**§3 の制約**を絶対の前提にする。
2. **§5 の最小プロトを1つ作る**（題材が未確定なら、まず「育つ庭」で仮置き。本人の題材選好が出たら差し替え）。
3. `first-steps` / `reflections` の localStorage を読み書きして、現行の積み重ねと連動させる（§4）。
4. **気楽さを最優先で検証**：起動の速さ、1手30秒、責めない設計。重くなったら機能を削る。
5. **題材・行動フックなど“刺さるか”に関わる判断は、勝手に確定せずコア対話に戻す。** あなた（実装側）はプロトを出す、刺さるかの判断はユーザー＋コア対話が持つ。

### 10.1 作って当てたもの（事実）

- **プロトA：育つ庭（idle・低操作）** … §5 の素直な実装。`first-steps`/`reflections` 連動、咲く/連続日数。
  - 本人反応（事実）：**「もう少しやりがいが欲しい」「進んでいく感じがしない」**。引力が弱く、却下。
- **プロトB：パズドラ風パズル×バトル（フロアを上る）** … ゲーム性・進行感を足した版。今日の一歩でその回を強化。
  - 本人反応（事実）：**「めちゃくちゃ微妙、これならやらない」**。却下。

### 10.2 本人発言から取れた事実（重要・設計の土台を更新する）

1. **疲れてIGに伸びる瞬間に欲しいのは「受け身でぼーっと眺める」**こと。＋「もしかしたら良い物（いい百均紹介・楽しいイベント等）に出会えるかも」という淡い期待。
   → 求めているのは **ダラ見（受動消費）＋小さな発見**。能動的な「ゲーム」ではない。
2. **求めているのは“新しさ”**。本人いわく「過去のを見たくなるなら、大量の写真をとっくに見返してるはず。でもしない」。
   → **自分の過去/痕跡を見返す行動は、本人はしない**（写真で実証済み）。

### 10.3 これが崩した前提（コア対話で要再検討）

- **§2・§3④「痕跡が育つ＝見える＝報酬になる」という前提に赤信号。**
  本人は自分の過去（写真）を見返さない。「積み上がりを見せれば嬉しいはず」が成り立たない可能性が高い。
  → 「痕跡の可視化」を報酬の中心に据える設計は、見直しが要る。
- **「ふり返りが進むゲーム」という当初フレーム自体がズレている可能性。**
  本人の“逃げ込み”動作は能動的ゲームではなく受動的ダラ見。ゲーム化は方向違いかもしれない。

### 10.4 核心の矛盾（＝§6「最大の不確実性」の正体）

- **欲しい**：受け身・**新しい**・外からの刺激（＝ほぼ IG そのもの）
- **やらない**：努力（ゲーム）／自分の過去を見返す
- **ゴールが要求**：痕跡・ふり返り（＝過去・自分・内向き）

→ **「新しさ（外向き）」と「ふり返り（内向き・過去）」はベクトルが真逆。** ここをどう接ぐかが全課題。

### 10.5 残っている筋（仮説・コア対話で選ぶ）

- ❌ 自作コンテンツで IG の新しさに勝つ（§6どおり負け確）。
- ❌ 自分の痕跡を見返させる（10.2-2 で本人否定）。
- ⭕ 仮説A：**本当に新しい外部刺激（イベント/百均/面白ネタ等）を“借りて”流し込み、見る“ついで”に痕跡が一滴だけ残す**。コンテンツは作らず借りる、差別化は「痕跡が残る」一点。※設計の根を変える。要：コンテンツ供給源、ふり返りをどう“ついで”に忍ばせるか。
- ❓ 仮説B：**“疲れてIGの瞬間”の置き換えは諦め、痕跡・積み上げは別タイミングで取りにいく**戦略転換。

### 10.6 コア対話で決めるべき問い（§7 の更新）

1. 当初の「カジュアルゲーム」フレームを維持するか、**「受動ダラ見＋発見」型**に切り替えるか。
2. 「痕跡の可視化＝報酬」をどこまで諦める／作り替えるか（10.3）。
3. 「新しさ」をどこから供給するか（外部を借りる／生成する／そもそも狙わない）。
4. **ふり返りを“見返させる”のではなく、どう受動消費の中に忍ばせるか**（本人は見返さない前提で）。
