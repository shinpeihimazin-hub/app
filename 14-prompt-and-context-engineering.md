# プロンプト・コンテキストエンジニアリング（エージェント品質の土台）

エージェントの精度は、モデルやフレームワークより **プロンプト（指示）とツール記述（ACI）とコンテキスト管理** で決まることが多い。公式ベストプラクティス（`platform.claude.com/docs/.../prompt-engineering`）に基づく実務リファレンス。フェーズ4（成果物生成）で使う。

> 具体プロンプト例は最新モデル向けの公式ガイダンスに準拠。モデル世代で最適な言い回しは変わるため、実装時に公式で確認する。過剰な「CRITICAL/MUST」は最新モデルでは過剰発動を招くので、通常の指示（"Use this tool when..."）にする。

---

## 1. システムプロンプトの基本（適用順）

1. **明確・直接的に。** 曖昧さを消す。「提案して」ではなく「変更して」。行動させたいなら行動を指示する（最新モデルは指示追従が精密）。
2. **役割を与える。** system prompt でペルソナ/専門性を定義（「あなたはシニアなセキュリティレビュアー」）。
3. **文脈を足す。** なぜこのタスクか、出力が誰にどう使われるかを書くと質が上がる。
4. **例を使う（multishot）。** 望む入出力の例を1〜数個。特に出力形式は具体例で固定する。
5. **XMLタグで構造化。** `<instructions>` `<context>` `<example>` 等で領域を区切ると解釈が安定。
6. **考えさせる（thinking）。** 複雑な多段推論やツール結果の反省には thinking を使わせる（§4）。
7. **出力形式を厳密に指定。** JSONならスキーマと例を添える。
8. **複雑なら分割（プロンプトチェイニング）。** 中間出力を検査/評価したいときは別呼び出しに分ける。

### 行動の積極度を制御する
```
<default_to_action>
既定で、提案ではなく実装する。ユーザーの意図が不明なら、最も有用そうな行動を推測して進め、
不足はツールで調べる（当てずっぽうで埋めない）。
</default_to_action>
```
逆に慎重にさせたいなら「明示的に指示されるまで実装・変更しない。曖昧なら情報提供と推奨に留める」。

## 2. ツール記述（ACI）— ここが弱いと賢いモデルでも失敗する

Anthropicの中核主張: **ツールのインターフェースとドキュメントに、人間向けUIと同じだけ投資する。**

- **`description` に「何を・いつ・何を返すか」を具体的に書く。** これがモデルの呼び出し判断の主材料。
- **入力スキーマを厳密に（JSON Schema）。** 必須/enum/範囲を明示。曖昧な自由文字列を避ける。`strict: true` でスキーマ厳密準拠。
- **紛らわしいツールを統合/整理する。** 似た機能の複数ツールは誤選択を招く。少数の明快なツールに寄せる。
- **意味のある結果を返す。** 生のIDや切り詰めた出力でなく、モデルが次に使える形（要点＋必要な識別子）。
- **エラーは「次にどうすべきか」を導く形で返す。** 例外で落とさず、`is_error` で「何が起きて、どう直すか」を返す（[`11`](./11-claude-code-tool-use-and-sdk.md)）。
- **トークン効率。** ツール定義も結果も毎ターンのコスト。冗長を削り、多数なら[tool search](./11-claude-code-tool-use-and-sdk.md)で遅延ロード。
- **ツールもテスト・評価する。** 実際にモデルに使わせて、誤用・失敗を観察して記述を直す（[`06`](./06-evaluation-and-iteration.md)）。

## 3. 並列ツール呼び出し（速度・効率）

独立したツール呼び出しは並列化できる。デフォルトで高成功率だが、プロンプトで最大化できる。
```
<use_parallel_tool_calls>
依存の無い複数ツールは並列で呼ぶ。例: 3ファイル読むなら3呼び出しを同時に。
依存がある（前の結果が次の引数になる）場合のみ逐次。プレースホルダや当て推量の引数は禁止。
</use_parallel_tool_calls>
```
- 副作用の無い読み取り系に `readOnlyHint: true` を付けると並列化されやすい（[`11`](./11-claude-code-tool-use-and-sdk.md)）。
- 逆に安定重視で逐次化したいなら「各ステップ間に小休止を入れ逐次実行」。

## 4. 思考（thinking / effort）の使い方

- 最新モデルは **adaptive thinking**（`effort` とクエリ複雑度で思考量を自動調整）。多段ツール使用・複雑コーディング・長期ループなどエージェント的作業で有効。
- **ツール結果の後に反省させる**と質が上がる:
  ```
  ツール結果を受け取ったら、その質を吟味し、最適な次手を決めてから進む。
  思考を使って計画・反復し、最善の次の行動を取る。
  ```
- **考えすぎ対策**: 最新モデルは高effortで過剰探索することがある。「1つのアプローチを選んで貫け。矛盾する新情報が出るまで決定を蒸し返すな」。ハード上限が要るなら `effort` を下げる／`max_tokens`。
- **過剰発動対策**: 「CRITICAL: 必ず使え」等の強い言い回しは最新モデルで過剰発動を招く。「〜のとき使う」に落とす。

## 5. 長期タスクのコンテキスト管理（エージェントの持久力）

複数コンテキストウィンドウにまたがる長時間タスクの要。**状態を外部に保存し、少しずつ確実に進める。**

- **構造化データはJSON、進捗メモは自由文。** テスト状況等は `tests.json`、一般進捗は `progress.txt`。
- **gitで状態追跡。** 何をしたかのログ＋復元可能なチェックポイント。最新モデルはgit活用が得意。
- **増分主義。** 一度に全部やらせず、少数を確実に進めさせ、進捗を書かせる。
- **圧縮を前提に早期終了を防ぐ**（Claude Codeのように圧縮/外部保存がある場合）:
  ```
  コンテキストは上限接近時に自動圧縮され、続きから作業できる。
  トークン予算を理由に早期終了するな。予算接近時は現在の進捗と状態をメモリに保存してから続けよ。
  ```
- **新規ウィンドウで再開 vs 圧縮**: 最新モデルはローカルファイルから状態復元が得意。圧縮より新規ウィンドウ＋「pwd確認／progress.txt・tests.json・git log を読め／基本の統合テストを通してから次へ」の方が良い場合がある。
- **検証手段を与える**: 自律タスクが長くなるほど、人間なしで正しさを検証する手段（Playwright MCP・テストスイート・computer use）が要る。
- **初回ウィンドウは別プロンプト**: 最初にテスト作成/セットアップスクリプト作成で枠組みを作り、以降はtodoを反復。「テストの削除・改変は禁止（機能欠落・バグにつながる）」と明示。

## 6. 自律と安全（取り返しのつかない操作）

ガイダンス無しだと、削除・force-push・外部投稿など不可逆操作を取ることがある。
```
行動の可逆性と影響を考えよ。ファイル編集やテスト実行などローカルで可逆な行動は推奨。
だが不可逆・共有システムに影響・破壊的な行動は、実行前にユーザーに確認せよ。
確認が要る例:
- 破壊的: ファイル/ブランチ削除、DROP TABLE、rm -rf
- 不可逆: git push --force、git reset --hard、公開済みコミットの改変
- 他者に見える: push、PR/issueへのコメント、メッセージ送信、共有インフラ変更
障害に当たっても破壊的行動をショートカットにするな（--no-verifyで安全チェックを迂回しない等）。
```
確定的に強制したい安全制約は、プロンプトでなく[フック](./12-claude-code-hooks-and-plugins.md)（`PreToolUse` deny）で。安全設計全体は [`08 §6.5`](./08-agent-primitives-and-composition.md)。

## 7. よくある失敗モードと対処

| 失敗モード | 対処 |
|---|---|
| **過剰実装（overengineering）** | 余計なファイル・抽象・未要求の柔軟性を作る癖。「解を最小に保て。要求されていない抽象や設定可能性を足すな」 |
| **サブエージェント乱用** | 直接grepで済む探索にまでサブエージェントを spawn。「並列化・独立文脈・独立ワークストリームの時だけサブエージェント。単純/逐次/単一ファイル/文脈共有が要る時は直接やれ」 |
| **一時ファイルの散らかし** | 「反復用に作った一時ファイル/スクリプトはタスク終了時に削除せよ」 |
| **テスト通過に固執・ハードコード** | 「テストを通すことでなく、根本の機能を正しく実装せよ。テストの削除・改変は禁止」 |
| **ハルシネーション（コーディング）** | 検証手段（テスト/型/実行）を与え、根拠を確認させる。存在しないAPIを使わせない |
| **調査が浅い/偏る** | 「構造的に調べ、複数の競合仮説を立て、確信度を進捗メモに記録し、定期的に自己批判せよ」 |

## 8. プロンプトチェイニング（自己修正）

最新モデルは多段推論を内部で扱うが、**中間出力を検査/評価/分岐したい時**は明示的に別API呼び出しに分ける。最頻出は自己修正: **下書き生成 → 基準に照らしてレビュー → レビューに基づき修正**。各ステップを別呼び出しにすればログ・評価・分岐できる（= [`13`](./13-advanced-agent-patterns.md) のエバリュエーター・オプティマイザー）。
